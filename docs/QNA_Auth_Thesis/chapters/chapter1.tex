\section{Background}
Authentication systems are usually built around passwords, tokens, or certificates. These mechanisms are practical and well understood, but they also create familiar weaknesses such as reuse, theft, replay, and dependence on explicit secrets. A complementary line of thought in security research is to ask whether physical characteristics of a device can act as a verification signal. This idea appears in hardware security, biometrics, \ac{puf} literature, and behavioral identity systems. QNA-Auth studies a closely related question: can stochastic sensor noise from a device's camera and microphone be processed into a repeatable signal for high-confidence device matching?

The project approaches the problem as one of similarity rather than identity. This distinction is critical. A raw or learned embedding is not treated as a password, token, or permanent secret. Instead, it is treated as a biometric-like template that is useful only when fresh samples from the same measurement path can be mapped into a nearby representation. This project therefore avoids claiming that device noise is a cryptographically secure identifier. It claims only that, under controlled conditions, it may provide a useful verification cue.

The background of the work combines signal processing, \ac{ml}, and application security. Sensor noise is naturally stochastic and environment-sensitive, so the system cannot rely on exact equality of repeated measurements. Instead, it extracts engineered statistical and spectral features, maps them into a learned embedding space, and measures closeness by cosine similarity. The full pipeline is then wrapped in a web service with database-backed persistence, challenge expiry, rate limiting, and audit logging. The result is a capstone project that is not only model-driven but system-driven.

\subsection{Overview of the Problem Domain}
Every camera sensor and microphone path introduces subtle acquisition artifacts. Cameras capture fixed-pattern noise, readout inconsistencies, quantization effects, and noise traces shaped by the device pipeline. Microphones expose electrical noise, gain path characteristics, ambient capture artifacts, and hardware response differences. None of these traits alone is guaranteed to be unique or stable across all settings, but they may still produce a measurable structure that can separate same-device and different-device samples with non-trivial accuracy.

In a research setting, this creates an attractive problem: the signal is noisy, the classes are close, and the security context forces careful thinking about failure modes. The problem cannot be solved credibly by a simple one-shot classifier. It requires repeated sampling, feature stability, calibrated thresholds, and a clear attacker model.

\subsection{Current Systems and Existing Solutions}
Most deployed systems still rely on static credentials or hardware-bound cryptographic modules. They are effective, but they solve a somewhat different problem: explicit identity proof rather than latent physical similarity. Biometric systems, device fingerprinting tools, and behavioral analytics come closer to the problem studied in this thesis, but they often require richer signals or operate at a much broader scale. QNA-Auth is narrower. It focuses on camera and microphone noise and asks whether a modest but complete software system can use those signals for prototype-level verification.

\subsection{Limitations of Existing Approaches in the Project Context}
Three limitations motivate the present work. First, purely secret-based methods do not test whether a device's own measurement path can contribute to verification. Second, many small academic prototypes stop at a model notebook and do not become a usable application. Third, some experimental systems overstate their results by collapsing uncertainty into a binary pass/fail output. QNA-Auth responds to these limitations by building an end-to-end stack and using confidence bands, source-profile guards, and attacker-aware documentation.

\section{Motivation}
The project was undertaken for both academic and practical reasons. Academically, it allows integration of signal processing, metric learning, threshold calibration, and secure system design in one coherent thesis. Practically, it produces a live demo system with backend APIs, frontend screens, enrollment/authentication flows, and review-ready artifacts. It is therefore useful not only as a research exercise but as a demonstration of software-engineering rigor.

There is also a methodological motivation. A good capstone should resist the temptation to claim more than it can defend. The QNA-Auth repository already reflects this discipline by repeatedly stating that the system performs ``high-confidence device matching'' rather than full identity proof. Preserving that wording is not rhetorical caution; it is part of the project's technical correctness.

\section{Scope of the Project}
The project scope includes multi-source enrollment, runtime authentication, challenge generation, challenge verification, device listing, device metadata retrieval, and device deletion. The active runtime sources are camera and microphone. Training and evaluation scripts, a review brief generator, diagnostic tools, and a React-based interface are also part of the implemented scope.

The project does not attempt to solve broad deployment calibration, hardware attestation, universal cross-device uniqueness, or adaptive spoofing resistance. It is also not a replacement for cryptographic credentials. The appropriate deployment interpretation is that of an auxiliary verification prototype.

\section{Need for the Proposed System}
The proposed system is needed because it addresses a practical gap between proof-of-concept signal analysis and usable secure software. It transforms sensor-noise experimentation into a structured service. The need is especially visible in four places:
\begin{itemize}
	\item there is a clear enrollment workflow rather than ad hoc scoring,
	\item there is database-backed storage of device metadata and challenges,
	\item there is security hardening against replay through nonce binding and \ac{hkdf},
	\item there is a reviewer-safe evaluation and demo narrative grounded in actual repository artifacts.
\end{itemize}
