\section{System Architecture}

\subsection{Architecture}

A layered architectural approach which separates each major function (data collection, data preprocessing, data embedding, application logic, data persistence, user interface) into individual layers was used in the project to guarantee modularity, scalability, and ease of maintenance.

At the startup, the backend initializes the model, preprocessing pipeline, and database connections to ensure efficient requests handling.

\subsection{System Design}

\begin{figure}[H]
	\centering
	\includegraphics[width=0.95\linewidth]{images/system_design.png}
	\caption{System design pipeline of QNA-Auth}
	\label{fig:system_design}
\end{figure}

The design of this system has followed a structured approach; which consists of data capture, feature engineering model processing and decision logic. Input from the camera and microphone (e.g. sensor data) is processed into statistical and frequency-based vectors; then transformed using normalization techniques. Outputs from the camera and microphone are then sent through a Siamese network to produce embeddings; from which a set of similarity scores are derived. Finally, the process includes weighted fusion and thresholding to provide a robust; reliable device authentication outcome.

\section{Detailed Design}

\subsection{Class Diagram}

\begin{figure}[H]
	\centering
	\includegraphics[width=0.95\linewidth]{images/class_diagram.png}
	\caption{Class diagram of QNA-Auth system}
	\label{fig:class_diag}
\end{figure}

The class diagram shows how the system is organized into modular components and how those components interact with one another. The entry point is through the FastAPI server, which coordinates both the enrollment and authentication workflows. Several classes such as DeviceEnroller, DeviceAuthenticator, and NoisePreprocessor are responsible for processing data, extracting features, and performing verification logic. The design has created a clear separation of concerns, allowing the system to be easily scalable, maintainable, and extensible.

\subsection{API Design}

The backend exposes a small number of endpoints that are in alignment with the workflow of the system:

\begin{itemize}
	\item \texttt{POST /enroll}
	\item \texttt{POST /authenticate}
	\item \texttt{POST /challenge}
	\item \texttt{POST /verify}
	\item \texttt{GET /devices}
	\item \texttt{GET /devices/\{device\_id\}}
	\item \texttt{DELETE /devices/\{device\_id\}}
	\item \texttt{GET /health}
	\item \texttt{GET /stats}
\end{itemize}

This API structure enables enrollment, authentication, device management, and system monitoring while keeping an easy to understand and use structure.
