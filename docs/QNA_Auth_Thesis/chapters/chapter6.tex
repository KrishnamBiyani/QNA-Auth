\section{Backend Implementation}
In the server/app.py file, we have the entry point for our backend system. Here, we set up our FastAPI application and enable cross-origin settings, load model databases (also known as checkpoints), set up the feature converter, provide access to the database, and create the set of services required at runtime (enrollment, authentication, and handling of challenges).

By using a startup model approach for our API, handler functions only execute business logic, meaning that they will not have to re-create any models. These are built using Pydantic request and response objects, which is important for ensuring that the application enforces field requirements, types/constraints on fields, and shape validations of the returned payload (e.g. samples sent by clients).

Furthermore, Pydantic request and response definitions will lead to improved quality of OpenAPI documentation generated by our backend application and will be available via the /docs endpoint.

\section{Enrollment Module}
Auth/enrollment.py has the enrollment implementation, where it creates device IDs, gets/normalizes raw samples, transforms them into features, generates source embeddings, builds template banks, saves template bundles, and records metadata. Metadata contains the following:
\begin{itemize}
	\item Device name,
	\item Number of samples,
	\item Active sources,
	\item Feature dimension,
	\item Source profiles,
	\item Source weights,
	\item Template strategy and template counts,
	\item Drift model parameters,
	\item Attacker-model notes.
\end{itemize}

This is one of the stronger parts in the overall implementation because having metadata makes each enrollment interpretable and easily reviewable.

\section{Authentication Module}
The file auth/authentication.py is where the implementation of Authentication is housed. In this file, the source of authentication is validated, runtime profiles are computed, matches are checked against possible mismatches, runtime embeddings are created, scores are generated against stored template banks and confidence-band plus margin logic is applied. The module allows for rolling re-enrollment after strong matches have been identified in the case of drift updates being enabled.

One special note in the development of the module was to use a top-k scoring methodology when comparing the scores in the template banks. Rather than using just a single best or a single average score from the matching templates, the module creates an average score of the top x number of template similarities. This provides less influence on the score of one outlier template.

\section{Challenge-Response Module}
The implementation of the challenge-response logic can be found in auth/challenge response.py. A challenge is made of the following properties: a unique challenge ID, the identifier of the device that will respond to the challenge, a randomly generated nonce, a creation time, and an expiration time. The stable bytes of an embedding are built by taking the coarse quantization of the embedding, and then appending the nonce and server secret to those bytes to compute an MAC key. As a result, the final response will be an over-challenge data. This will help in preventing the straightforward reuse of a challenge because challenges are created once (single use) and each nonce is unique to a specific challenge.

\section{Database Implementation}
SQLAlchemy is used to implement the persistence layer in db/models.py. Three tables are created:
\begin{itemize}
	\item \texttt{Device}: holds the following attributes: Device ID, name, embedding path, metadata, timestamp.
	\item \texttt{Challenge}: holds the state of an active challenge and the time it will expire.
	\item \texttt{AuditLog}: holds information about the events that take place when performing important actions.
\end{itemize}

The chosen database is SQLite (as the default), which will work fine as a database for a university capstone prototype project and makes setting up the database easier than other databases and limiting the schema results in less operational overhead when maintaining.

\section{Frontend Implementation}
The frontend app can be found under the frontend/src/ directory and provides a simple yet effective control surface. The home page describes the project as a "Verification of Noise Based Devices via Audio/Camera Fingerprints". There are dedicated pages for enrollment, authentication and device viewing. The frontend service layer centers around making HTTP calls from the frontend to the backend, while providing a set of typed request and response interfaces.

\section{Configuration and Operational Controls}
Implementing configurable parameters within the config.py file enables implementation flexibility while adhering to an appropriate level of engineering complexity. Configuration variables of interest are:
\begin{itemize}
	\item Model checkpoint file paths
	\item Confidence Thresholds
	\item Identification margin
	\item Source weights
	\item Drift update
	\item Expiration of challenges
	\item Demo mode source limits
	\item Up to and including optional API key rate limiting
\end{itemize}

As these parameters are configured, they can be used as tools in a thesis that would allow for experimentation in a well-defined way while still allowing for re-experiencing similar experiments using the same set up.

\section{Illustrative Code Snippet}
Here is an illustration of the implementation style by looking at a configuration block that represents what can be done with this approach to creating a configuration:

\begin{lstlisting}[language=Python]
AUTH_CONFIDENCE_STRONG = 0.97
AUTH_CONFIDENCE_UNCERTAIN = 0.92
AUTH_IDENTIFICATION_MARGIN = 0.02
AUTH_TEMPLATE_CHUNK_SIZE = 5
AUTH_MAX_TEMPLATES_PER_SOURCE = 4

AUTH_SOURCE_WEIGHTS = {
    "camera": 0.7,
    "microphone": 0.3,
}
\end{lstlisting}

This conveys the entire philosophy of how the implementation will be performed with respect to making all run-time rule representations visible and easy to review versus embedding key decision points elsewhere in opaque programming constructs.
